# Clarify Workflow

Use this workflow to identify the real question before trying to solve it.

## When to use

Use this workflow when:

- the user's question is vague
- key terms are undefined
- the user is mixing goal and method
- a tactical question is hiding a strategic one
- the problem is emotionally charged or poorly framed
- the issue cannot be examined because nothing has been defined cleanly

## Early exit

If the question is already specific, measurable, and well-framed, skip this workflow. Proceed directly to Deconstruct or Decide.

## Minimum information

If the input is too vague to analyze (no discernible topic, no context, no decision), ask 1-2 focused questions before proceeding:
- "What are you trying to decide or achieve?"
- "What triggered this question right now?"

Do not run all five steps on an empty input.

## Core question

What is the user actually trying to decide, understand, or achieve?

## Process

### Step 1: Restate the surface question

Write the question as the user currently frames it.

### Step 2: Identify vague or overloaded language

Look for terms such as:

- better
- professional
- scalable
- necessary
- good
- worth it
- risky
- simple
- successful
- high quality

Ask what each one means in this specific context.

### Step 3: Separate goal from method

Check whether the user is naming:

- the real outcome
- a preferred method
- a fear
- a constraint
- a proxy metric
- a social expectation

Do not assume the user's preferred method is the real objective.

### Step 4: Surface initial assumptions

Identify the first layer of hidden assumptions in the framing.

Ask yourself:
- What is being treated as necessary that might not be?
- What would have to be true for the user's current framing to hold?
- What is being accepted without evidence?
- Which assumption is carrying the most weight?

Examples of common hidden assumptions:
- an app is necessary
- more complexity means more quality
- competitors define the right structure
- the bottleneck is already known
- the current approach is the only viable one
- this must be done now

### Step 5: Rewrite the problem cleanly

State the sharpest version of the actual question. A sharp question:
- names a specific decision, outcome, or constraint
- is answerable (not a vague aspiration)
- does not contain any of the vague terms identified in Step 2

If there are multiple legitimate interpretations, state the top 2-3 candidate reframes ranked by likelihood. Proceed with the most likely one and note the alternatives.

## Output format

- Surface question
- Ambiguous terms
- Hidden assumptions
- Real goal
- Reframed question

## Handoff

Pass the Real Goal and Reframed Question to the next workflow (typically Deconstruct). Carry forward any unresolved ambiguities or unverified assumptions.

## Self-check

Before outputting, verify:
- Is the reframed question narrower and more precise than the surface question?
- Does it still address what the user actually cares about?
- Have I projected my own reframe onto a problem the user did not actually have?

## Quality bar

A good clarification should make the problem narrower, more precise, and easier to evaluate.
